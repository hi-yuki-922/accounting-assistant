## Context

商品信息管理（Phase 1A）即将落地，客户管理已存在。本模块在现有分层架构上新增订单能力，核心挑战在于：
- 订单与记账记录的双向关联（order.accounting_record_id + accounting_record.order_id）
- 结账时的事务一致性：订单状态变更 + 记账记录生成需在同一个事务中完成
- 订单明细的商品名称冗余存储（商品可能改名或删除，订单应保留历史快照）
- 散客场景（customer_id 可空）与固定客户场景并存

现有技术约束：
- 实体 ID 采用日期序列模式，需配套 seq 实体
- 逻辑外键，无物理 FK 约束
- Sea-ORM entity-first，无 migration 文件
- accounting_record 已有字段：id, amount, record_time, accounting_type, title, channel, remark, write_off_id, create_at, state, book_id

## Goals / Non-Goals

**Goals:**
- 实现完整的订单生命周期：创建 → 编辑（结账前） → 结账/取消
- 销售订单结账时自动生成 Income 记账记录
- 采购订单结账时自动生成 Expenditure 记账记录
- 支持应收/实收双金额（覆盖抹零、让利场景）
- 支持散客（customer_id 可空）
- 订单与记账记录双向可查询
- 订单编号简洁直观（`#N` 格式），适合口头沟通
- 支付渠道在结账时选择，符合真实业务流程
- 前端支持卡片/列表双视图，适应不同查看需求

**Non-Goals:**
- 不实现退款流程（Phase 2 考虑 Cancelled 状态即可，不实现部分退款）
- 不实现库存联动（Phase 2）
- 不实现多币种
- 不实现 Agent 自然语言集成（Phase 1C）
- 不实现订单类型和客户的修改（创建后锁定）

## Decisions

### 1. 订单明细冗余存储商品名称

`order_item` 存储 `product_name` 和 `unit`，不仅依赖 `product_id`。理由：
- 商品可能改名或删除，订单应保留创建时的商品名称快照
- Agent 创建订单时可能遇到新商品（尚未录入系统），需要支持临时商品名
- 与 attachment 的 `master_id` 多态引用不同，此处冗余是必要的业务快照

### 2. 订单结账前可编辑（明细和备注）

订单在 Pending 状态下允许修改订单明细和备注，但不可修改订单类型（order_type）和客户（customer_id）。理由：
- 真实场景中，用户创建订单后经常需要调整明细（改数量、加商品、删商品）
- 取消重建的操作成本高，尤其是明细较多时
- 订单类型和客户是订单的核心属性，创建后应保持稳定
- 与记账记录的"未入账可修改"原则一致

采用通用 `update_order` 接口，传入订单 ID 和需更新的字段（明细列表、备注），服务层校验状态为 Pending 后执行更新。total_amount 和 actual_amount 根据更新后的明细自动重算。

**替代方案**：取消 + 重建 → 操作繁琐，明细多时体验差。

### 3. 结账操作在事务中完成

结账时在同一事务中执行：
1. 验证订单状态为 Pending
2. 创建 accounting_record（带 order_id）
3. 更新 order 的 accounting_record_id、status 为 Settled、settled_at 时间
4. 更新对应 book 的 record_count

**替代方案**：分步操作 + 补偿 → 过于复杂，事务是最简单可靠的方式。

### 4. accounting_record.book_id 的确定策略

订单结账生成记账记录时，book_id 设置为 None（即归入默认账本"未归类账目"）。
理由：
- 订单不属于任何特定账本，记账记录的账本归属应由用户后续手动调整
- 避免自动归属逻辑带来的耦合

### 5. 订单编号格式

`order_no` 采用当日序号格式：`#N`（如 `#1`, `#2`, `#3`）。理由：
- 简洁直观，适合商户口头沟通（"3 号单"）
- 当天内不重复，跨天可重复（通过 create_at 区分）
- 与内部 ID 解耦，ID 仍为日期序列模式 `YYYYMMDDNNNNN`
- `order_no` 的值直接取自 `order_seq` 的 `seq` 字段，拼接 `#` 前缀

**替代方案**：`ORD-YYYYMMDD-NNNNN` → 格式过长，口头沟通不便。

### 6. order_item 的 ID 使用自增

order_item 使用 SQLite 自增 ID，不使用日期序列。理由：
- order_item 是从属实体，不需要独立的全局唯一 ID
- 简化实现，与 attachment 的自增模式一致

### 7. 取消订单不删除记录

取消订单将 status 设为 Cancelled，不物理删除。理由：
- 保留审计轨迹
- 已取消的订单不应产生记账记录
- 与记账记录的 Posted 不可删除原则一致

### 8. 支付渠道移至结账阶段

订单创建时不选择支付渠道（`channel` 为 `None`），在结账时选择。理由：
- 符合真实业务流程——创建订单时只确定商品和数量，结账时才知道支付方式
- 减少创建订单时的操作步骤
- 结账时必须选择支付渠道（`SettleOrderDto` 中 channel 为必填）

### 9. 订单详情使用 Dialog 展示

订单详情不使用独立页面，而是在列表页中以 Dialog 形式弹出。理由：
- 不离开列表页，保持操作连贯性
- 桌面应用场景下 Dialog 比页面跳转更轻量
- 点击卡片或列表行即可弹出，关闭后上下文不丢失
- 可移除 `/orders/$orderId` 路由

### 10. 卡片/列表双视图设计

订单管理页面支持两种视图模式：

**卡片视图**（默认）：
- 仅展示今日订单，聚焦当日经营
- 两排独立 Tab 控制：类型 Tab（全部/销售/采购）+ 状态 Tab（全部/待结账/已结账/已取消）
- 两个 Tab 组合筛选（如"销售 + 待结账"）

**列表视图**：
- 展示全部历史订单，支持多维度筛选
- 筛选条件：时间范围、订单状态、金额范围、支付渠道、订单类型
- 支持分页，避免一次性加载大量数据
- 点击行打开详情 Dialog

### 11. 商品搜索覆盖多字段

创建/编辑订单时，商品搜索需覆盖商品名称、SKU 和关键词（keywords 字段）。理由：
- 关键词是用户为增强检索而手动标注的信息（如"六头;六粒;35#"）
- 覆盖多字段提高商品匹配率，减少搜索不到的情况
- 依赖 product-management 提案中的 keywords 字段

### 12. 订单明细单位不可修改

添加商品明细时，计量单位直接从商品信息中获取，不可手动修改。理由：
- 保证订单明细单位与商品信息一致
- 减少用户操作步骤（自动填充）
- 如果需要不同单位，应在商品信息中配置

## Risks / Trade-offs

- **[事务性能]** → SQLite 单写者模型下事务开销很低，桌面应用场景不存在并发瓶颈。
- **[订单编辑并发]** → 桌面单用户应用，不存在真正的并发编辑冲突。但仍需校验状态为 Pending 才允许编辑。
- **[book_id 归入默认账本]** → 所有订单产生的记账记录都进入"未归类账目"，商户需要手动归类。可通过后续的自动归类规则改善。
- **[accounting_record 新增字段]** → 属于 BREAKING 变更。entity-first 模式下 Sea-ORM sync 会自动添加新列，已有数据该列为 NULL，无迁移风险。
