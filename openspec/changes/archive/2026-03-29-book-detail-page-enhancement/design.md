## Context

当前账本详情页采用传统列表展示方式，仅支持检索和展示账本的记账记录。系统已具备基础的记账服务（添加、修改、过账记录），但在账本详情页的集成和用户体验层面存在不足。用户需要在账本详情页完成核心的记账记录管理操作，包括"记一笔"、编辑、删除、批量入账和冲账等功能。

现有的数据结构 `RecordWithCountDto` 仅包含原始记录和关联数量，不支持净金额显示和冲账详情的快速查看。主查询接口 `get_records_by_book_id_paginated` 使用多次查询的方式获取数据，存在性能优化空间。前端采用 shadcn/ui 组件库，支持现代的交互设计。

## Goals / Non-Goals

**Goals:**

1. **完整的记账记录管理功能**：在账本详情页提供"记一笔"、编辑未入账记录、删除未入账记录、单条入账、批量入账、冲账等核心操作，用户无需切换页面即可完成所有记账任务

2. **优化的数据展示和交互**：使用 HoverCard 按需加载方式展示冲账详情，净金额为 0 时使用灰色标记，提供清晰的数据可视化和快速访问

3. **高性能的数据查询**：使用 SQL JOIN 优化主查询接口，减少数据库查询次数，提升页面加载速度

4. **严格的数据验证和权限控制**：前后端双重验证确保数据一致性，状态权限控制确保操作合法性和可追溯性

5. **流畅的用户体验**：表格排序、批量操作确认、智能刷新策略、日期时间选择器等优化交互设计

**Non-Goals:**

- 不实现复杂的财务报表和分析功能（属于其他功能模块）
- 不修改现有的其他页面逻辑和组件
- 不引入重大的架构变更或重构（仅在现有架构上扩展）
- 不实现高级的离线或本地存储功能

## Decisions

### 1. 数据结构扩展

**决策**：扩展 `RecordWithCountDto` 数据结构，添加 `original_amount` 和 `write_off_records` 字段

**理由**：
- 需要同时保存原始金额和净金额用于显示和计算
- 提供冲账记录详情支持 HoverCard 按需加载
- 保持与现有结构的兼容性，前端可以逐步迁移

**替代方案考虑**：
- 方案 A：仅返回净金额，不包含原始金额和冲账记录 → ❌ 不支持 HoverCard 详情显示
- 方案 B：返回嵌套数据结构，包含完整的冲账层级 → ❌ 复杂度高，影响前端处理
- 方案 C：使用独立的查询接口获取冲账详情 → ✅ 选择此方案，平衡性能和复杂性

### 2. 按需加载 HoverCard 数据

**决策**：新增独立的 `get_record_write_off_details` 接口，在用户 hover 金额时按需调用

**理由**：
- 避免一次性加载大量冲账数据，提升主查询性能
- 减少不必要的网络请求，只加载用户实际需要的数据
- 支持数据缓存策略，避免重复请求

**替代方案考虑**：
- 方案 A：主查询一次性加载所有冲账数据 → ❌ 数据量大时影响性能
- 方案 B：使用预加载策略（hover 第一条记录时加载本页所有）→ ✅ 可选优化方案，后期可考虑
- 方案 C：使用 React Query 自动缓存 → ✅ 推荐结合使用

### 3. SQL JOIN 优化主查询

**决策**：使用 SQL LEFT JOIN 一次性获取原始记录和冲账数据，计算并返回净金额

**理由**：
- 减少数据库查询次数从多次（查询记录 + 查询关联数量 + 查询原始金额）到一次
- 利用数据库索引和查询优化器，提升性能
- 保证数据一致性，避免多次查询之间的数据变更

**替代方案考虑**：
- 方案 A：保持多次查询，使用应用层缓存 → ❌ 复杂度高，缓存一致性问题
- 方案 B：使用数据库视图 → ✅ 可选优化方案，但需要额外的数据库管理
- 方案 C：使用存储过程 → ❌ 增加维护成本，降低可读性

### 4. 净金额为 0 的特殊标记

**决策**：当净金额为 0 时，使用灰色文字标记并提供视觉区分

**理由**：
- 明确标识特殊情况，提醒用户注意
- 避免用户误认为数据错误或异常
- 保持界面的一致性和美观性

**替代方案考虑**：
- 方案 A：不进行特殊标记，正常显示 → ❌ 用户可能忽略特殊情况
- 方案 B：使用警告图标标记 → ✅ 可选方案，但可能影响界面简洁性
- 方案 C：使用特殊样式（如虚线边框）→ ✅ 可选方案，增强视觉效果

### 5. 排序基准选择

**决策**：金额排序基于净金额，而非原始金额

**理由**：
- 净金额是用户实际看到的数值，按此排序更符合用户预期
- 净金额反映了原始记录和所有冲账的综合结果，更准确的财务表示
- 保持显示和排序的一致性

**替代方案考虑**：
- 方案 A：按原始金额排序 → ❌ 与显示不一致，可能造成混淆
- 方案 B：支持切换排序基准 → ✅ 可选方案，提供更多灵活性
- 方案 C：金额字段不支持排序 → ❌ 限制用户的使用场景

### 6. 批量入账事务策略

**决策**：采用后端预验证 + 事务保证原子性，全部成功或全部失败

**理由**：
- 前端复选框限制（仅待入账记录可勾选）避免大部分错误情况
- 后端预验证快速失败，避免不必要的事务开销
- 事务保证数据一致性，避免部分成功导致的数据混乱

**替代方案考虑**：
- 方案 A：部分成功，跳过失败记录 → ❌ 数据一致性问题，用户难以理解结果
- 方案 B：重试失败记录 → ✅ 可选方案，但增加复杂性
- 方案 C：异步处理，后台执行批量操作 → ✅ 可选方案，提供更好的用户体验

### 7. 日期时间选择器实现

**决策**：使用 shadcn 的 DatePicker 选择日期，Input 输入时间，时间默认取当前时间

**理由**：
- DatePicker 提供良好的视觉体验和交互设计
- Input 支持标准的 HTML5 time 类型，兼容性好
- 分别处理日期和时间，提供更灵活的控制

**替代方案考虑**：
- 方案 A：使用原生 datetime-local input → ❌ 浏览器兼容性不一致，样式控制困难
- 方案 B：使用第三方库 react-datetime-picker → ✅ 功能完整但需要引入新依赖
- 方案 C：使用 Calendar + 自定义时间选择组件 → ✅ 推荐方案，平衡功能和依赖

### 8. 复选框限制逻辑

**决策**：不可勾选的记录设置 disabled 状态置灰不可选择，仅待入账记录可以勾选

**理由**：
- 用户明确知道哪些记录可以勾选，避免操作错误
- 保持视觉一致性（所有记录都有复选框）
- 符合常见的 UI 设计模式，降低学习成本

**替代方案考虑**：
- 方案 A：不显示复选框，只显示操作按钮 → ❌ 不支持批量操作
- 方案 B：点击不可勾选的记录时弹出提示 → ✅ 可选方案，提供更多指导
- 方案 C：隐藏不可勾选的记录的复选框 → ❌ 视觉不一致，可能造成混淆

### 9. 数据刷新策略

**决策**：添加/删除/批量入账后跳转到第一页并刷新列表，编辑/冲账后保留当前页和筛选条件并刷新列表

**理由**：
- 新增记录通常希望在列表顶部显示
- 删除/批量操作影响当前页数据结构，跳转更合理
- 编辑/冲账操作不影响页面整体结构，保留用户上下文更友好

**替代方案考虑**：
- 方案 A：所有操作都跳转到第一页 → ❌ 编辑/冲账时用户丢失当前上下文
- 方案 B：所有操作都保留当前页和筛选条件 → ❌ 删除/批量操作可能导致空页或数据不一致
- 方案 C：智能刷新，根据操作类型自动选择刷新策略 → ✅ 推荐方案，更好的用户体验

### 10. WriteOff 类型和金额规则

**决策**：新增 WriteOff 类型，冲账金额可以输入正数和负数，记账类型固定为 WriteOff，记录状态固定为已入账

**理由**：
- 提供明确的业务语义，冲账是一种特殊的记账类型
- 支持灵活的金额处理（正数表示抵消，负数表示加强）
- 状态固定确保冲账记录的权威性和不可编辑性

**替代方案考虑**：
- 方案 A：不新增 WriteOff 类型，使用备注标记 → ❌ 语义不清晰，不利于查询和统计
- 方案 B：冲账金额固定为负数 → ❌ 不支持所有业务场景
- 方案 C：冲账记录状态可选，支持编辑 → ❌ 影响数据一致性和可追溯性

## Risks / Trade-offs

### 性能风险

**风险**：SQL JOIN 查询在数据量大时可能影响性能
**缓解措施**：
- 优化 SQL 语句，使用适当的索引
- 考虑分页优化，避免查询过多数据
- 监控查询性能，必要时进行进一步优化
- 考虑引入数据库查询分析工具

### 数据一致性风险

**风险**：前后端验证逻辑不一致可能导致数据问题
**缓解措施**：
- 清晰定义验证规则，确保前后端一致
- 在设计文档中明确说明所有验证逻辑
- 编写单元测试覆盖各种边界情况
- 进行端到端测试验证数据一致性

### 用户体验风险

**风险**：HoverCard 按需加载可能导致用户感知延迟
**缓解措施**：
- 实现数据缓存，避免重复请求
- 添加加载状态指示，提供明确的反馈
- 使用骨架屏占位，优化视觉体验
- 考虑预加载策略（hover 第一条时加载本页其他）

### 复杂度风险

**风险**：功能较多可能导致代码复杂度上升，维护成本增加
**缓解措施**：
- 采用组件化设计，分离关注点
- 遵循现有代码规范和架构模式
- 编写清晰的注释和文档
- 进行代码审查，保持代码质量

### 向后兼容性风险

**风险**：数据结构变更可能影响现有功能
**缓解措施**：
- 保持现有字段不变，仅添加新字段
- 不修改现有接口的返回格式，只进行扩展
- 提供清晰的迁移指南和兼容性说明
- 进行回归测试，确保现有功能不受影响
