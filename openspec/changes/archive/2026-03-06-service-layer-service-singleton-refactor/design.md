## Context

当前服务层使用 Rust 的 `OnceCell` 来存储单例实例。每个服务类（如 `AccountingRecordService`、`AttachmentService` 等）都有自己的 `instance()` 静态方法和 `init()` 初始化方法。服务在应用启动的特定时机通过调用这些静态方法进行初始化。

这种模式存在以下问题：
- 服务初始化时机分散在多个地方，难以统一管理
- 服务与 Tauri 应用生命周期脱节，缺乏明确的依赖关系
- 单元测试时需要模拟复杂的初始化流程
- 类型安全性依赖于运行时检查，编译时无法保证服务已初始化
- 服务之间的依赖关系难以表达和管理

## Goals / Non-Goals

**Goals:**
- 使用 Tauri 的 `app.manage()` 机制在应用构建阶段统一注册所有服务单例
- 通过 `State<'_, ServiceType>` 在命令处理函数中注入所需的服务
- 确保所有服务在第一个命令执行前已正确初始化
- 提供更清晰的依赖注入模式，编译时类型安全
- 简化单元测试中的服务创建和使用方式

**Non-Goals:**
- 不修改服务的业务逻辑实现
- 不改变数据库连接池的管理方式
- 不影响前端与后端的 API 接口
- 不改变服务之间的调用关系（只是改变服务获取方式）

## Decisions

### 1. 服务注册时机选择

**决策**: 在 Tauri app 构建阶段（`Builder::setup` 回调）使用 `app.manage()` 注册所有服务

**理由**:
- `app.manage()` 在应用启动时执行，确保所有服务在第一个命令执行前已初始化
- Tauri 保证 `State` 在整个应用生命周期内保持单例
- 可以按照正确的依赖顺序初始化服务
- 注册失败会阻止应用启动，避免运行时错误

**替代方案考虑**:
- 使用 `once_cell` + 懒加载：会导致初始化时机不可控，可能影响第一个命令的执行时间
- 使用 DI 框架：增加项目复杂度，Tauri 的 `State` 机制已足够

### 2. 服务方法签名调整

**决策**: 服务方法接受必要的 Tauri 上下文作为参数，而不是存储在服务单例中

**理由**:
- 保持服务类的无状态性，更易于测试
- 避免在服务中存储 Tauri 特定的上下文
- 更明确地表达方法对 Tauri 上下文的依赖

**示例**:
```rust
// 服务方法签名
pub async fn upload(&self, app: &AppHandle, path: String) -> Result<(), String> {
    // 使用 app 获取路径等信息
}
```

### 3. 命令处理函数参数模式

**决策**: 命令处理函数同时接受 `AppHandle` 和所需的 `State` 参数

**理由**:
- `AppHandle` 用于访问应用级功能和配置
- `State` 用于访问服务实例
- 保持参数列表的清晰和类型安全

**示例**:
```rust
#[tauri::command]
async fn upload_file(
    app: AppHandle,
    service: State<'_, AttachmentService>,
    path: String
) -> Result<(), String> {
    service.upload(&app, path).await
}
```

### 4. 测试策略调整

**决策**: 单元测试直接创建服务实例，不依赖 `OnceCell` 初始化

**理由**:
- 更简单的测试设置
- 避免复杂的初始化流程
- 可以模拟测试所需的状态

**示例**:
```rust
#[test]
fn test_service_method() {
    let service = AttachmentService::new();
    // 直接使用 service 测试
}
```

### 5. 错误处理策略

**决策**: 服务初始化失败会导致应用启动失败

**理由**:
- 服务是应用的核心依赖，缺少任何服务都会导致功能异常
- 失败快速比运行时崩溃更好
- 用户会在启动时收到明确的错误提示

## Risks / Trade-offs

**风险 1**: 大量命令函数需要修改，可能引入错误
→ **缓解**: 使用自动化工具辅助修改，逐步迁移，充分的测试覆盖

**风险 2**: 服务初始化顺序可能影响功能
→ **缓解**: 在 `lib.rs` 中按照依赖顺序明确初始化，通过编译器检查确保无循环依赖

**风险 3**: 需要调整现有的单元测试
→ **缓解**: 优先修改测试工具函数，批量更新测试代码

**风险 4**: 移除 `OnceCell` 可能影响某些需要懒加载的场景
→ **缓解**: 通过 `State` 提供的 `try_get()` 方法实现延迟访问，如果确实需要懒加载

**权衡**: 虽然需要修改大量代码，但长期收益包括更好的可维护性、类型安全性和可测试性
