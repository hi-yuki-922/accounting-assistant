## Context

### 当前状态

当前项目存在前后端枚举类型不一致的问题，导致以下后果：

1. **后端 (Rust)**:
   - 枚举定义：`AccountingType::Income`, `AccountingChannel::Cash` 等
   - `FromStr` 实现：从中文字符串解析（"收入" → `Income`）
   - `as_str()` 方法：返回中文字符串（`Income` → "收入"）
   - `Display` trait：返回英文枚举名称（`Income` → "Income"）
   - 数据库存储：中文字符串
   - IPC 序列化：英文字符串（通过 Serialize + Display）

2. **前端 (TypeScript)**:
   - 枚举定义：`enum AccountingType { Income = '收入', ... }`
   - 枚举值：中文字符串
   - IPC 调用：传递中文字符串
   - IPC 接收：期望中文字符串，实际收到英文字符串
   - 类型比较：`record.type === AccountingType.Income` 失败（"Income" !== "收入"）

3. **数据流问题**:
   ```
   前端(中文) → IPC → 后端解析(中文) ✓
   后端序列化(英文) → IPC → 前端接收(英文) ✗
   数据库写入(中文) / 读取(中文) / IPC返回(英文)
   ```

### 约束条件

1. **开发环境**：可以删除数据库重建
2. **Display trait**：必须保留，返回英文用于调试
3. **前端显示**：仅在前端显示时使用中文，IPC 传输始终使用英文
4. **向后兼容**：不涉及现有 API 的用户，但前后端内部是 Breaking Change

## Goals / Non-Goals

**Goals:**

1. 统一前后端枚举值为英文，确保 IPC 通信一致
2. 数据库存储英文枚举值，简化数据库操作
3. 保留 Display trait 用于调试日志，返回英文
4. 前端使用常量对象存储中文显示文本，实现显示逻辑与数据分离
5. 所有测试用例同步更新，确保测试通过

**Non-Goals:**

1. 不改变枚举的语义含义（收入仍然是 Income）
2. 不改变枚举的成员数量和类型
3. 不涉及国际化（i18n）框架的引入
4. 不影响其他服务的接口契约

## Decisions

### 决策 1: 枚举值使用英文标识符

**选择：** 使用 PascalCase 英文标识符作为枚举值

**理由：**
- 与 Rust 枚举成员名称一致（`Income`, `Expenditure` 等）
- 符合常见的枚举命名约定
- 便于类型安全和 IDE 自动补全
- 避免 JSON 序列化时的转义问题

**考虑的替代方案：**
- **snake_case**（如 `income`, `expenditure`）：与 JSON 习惯一致，但与 Rust 枚举命名风格冲突
- **UPPER_CASE**（如 `INCOME`, `EXPENDITURE`）：数据库常用风格，但 TypeScript 枚举不常用

### 决策 2: 前端使用常量对象存储显示文本

**选择：** 使用 `DISPLAY_TEXT` 常量对象存储枚举到中文的映射

**理由：**
- 显示逻辑与枚举定义分离，枚举值保持纯英文
- 便于维护和扩展（新增枚举成员时只需更新一处）
- 支持未来的国际化需求（可以替换为对象字典）
- 类型安全，编译时检查

**考虑的替代方案：**
- **使用函数映射**（如 `getLabel(enum)`）：运行时调用，但每次都需要 switch/case
- **枚举成员带中文属性**：无法满足 IPC 传输英文的需求

### 决策 3: 数据库完全重建而非迁移

**选择：** 删除现有数据库，让 SeaORM 重新创建表结构

**理由：**
- 开发环境，可以接受数据丢失
- SeaORM 的 schema builder 不支持数据迁移
- 重建数据库更快更简单，避免复杂的 UPDATE 语句
- 确保新架构的完整性

**考虑的替代方案：**
- **编写 SQL 迁移脚本**：更安全，但需要额外的维护成本

### 决策 4: 保留 strum::Display trait

**选择：** 保留 Display trait，返回英文枚举名称

**理由：**
- 用户明确要求保留
- 用于调试日志，英文更易读
- 自动实现，无需手动编写 `Display` trait
- 与 Serialize 行为一致

**考虑的替代方案：**
- **移除 Display trait**：更简洁，但失去调试便利性

### 决策 5: 修改 FromStr 从英文解析

**选择：** 从英文字符串解析枚举，匹配枚举成员名称

**理由：**
- 与 Display trait 的输出一致
- 简化序列化/反序列化逻辑
- 与 TypeScript 枚举值对齐
- 避免中文字符串在网络传输中的编码问题

**考虑的替代方案：**
- **使用数值序列化**：更高效，但损失可读性和调试便利性

## Risks / Trade-offs

### 风险 1: 数据丢失

**描述：** 删除数据库会导致所有现有数据丢失

**缓解措施：**
- 明确标注为开发环境操作
- 在设计文档中记录删除步骤
- 考虑提供数据导出功能（非本次变更范围）

### 风险 2: 前端显示遗漏

**描述：** 可能遗漏某些组件的枚举显示转换

**缓解措施：**
- 使用全局搜索确认所有枚举使用点
- 在 TODO 中列出已知的使用位置
- 编译时类型检查会捕获部分遗漏

### 风险 3: 测试覆盖不足

**描述：** 修改后某些测试用例可能无法覆盖新的枚举值

**缓解措施：**
- 系统性更新所有测试用例
- 运行完整测试套件验证
- 使用搜索工具确认所有硬编码的枚举字符串

### 风险 4: IDE 自动补全混淆

**描述：** 枚举成员名称和枚举值相同可能导致理解混淆

**缓解措施：**
- 在代码注释中明确说明枚举值是字符串
- 提供类型提示和文档注释
- 使用 TypeScript 的 enum 类型系统保证类型安全

### 权衡 1: 可读性 vs 类型安全

**描述：** 英文枚举值对中文用户不够直观

**权衡：** 牺牲部分可读性，换取类型安全和数据一致性
**缓解措施：** 前端显示文本使用中文，保证用户体验

### 权衡 2: 实现复杂度 vs 架构清晰度

**描述：** 需要维护前端的显示文本映射对象

**权衡：** 增加少量维护成本，换来架构更清晰
**缓解措施：** 使用 TypeScript 对象字面量，类型推导自动工作
