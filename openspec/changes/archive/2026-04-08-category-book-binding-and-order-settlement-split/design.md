## Context

当前订单结算时，`settle_order` 将所有记账记录硬编码写入默认账本（`DEFAULT_BOOK_ID = 10000001`，"未归类账目"）。商品实体（`product`）与账本实体（`accounting_book`）之间没有任何关联，订单项（`order_item`）也没有品类维度的记账路由。

实际业务中，不同品类的商品（贝类、鱼类等）需要记账到对应品类的销售账本或进货账本，且一张订单经常包含多种品类商品。同时，订单上的折扣/抹零需要以冲账形式准确反映在各品类账本中。

当前数据流：
- 商品 `category` 字段为自由文本字符串，无实体关联
- `product.Relation` 为空枚举，无任何外键关系
- `order.accounting_record_id` 为单条 FK，只能关联一条记账记录
- `AccountingType` 枚举仅有 `Income`、`Expenditure` 两种类型

## Goals / Non-Goals

**Goals:**
- 建立商品品类实体，作为商品的基础资料，支持品类与账本的绑定
- 订单结算时按品类自动拆分记账，每张品类记账记录归入对应账本
- 折扣/抹零以冲账（WriteOff）形式记录，与主记录归入同一账本
- 订单新增业务类型字段，用于后续统计查询
- 采购订单 UI 文案准确反映"应付/实付"

**Non-Goals:**
- 不实现品类层级的树形结构（如大类-小类），当前仅需一级品类
- 不支持多收场景（实收 > 应收），冲账金额始终为负数
- 不改变订单项（order_item）的数据结构
- 不实现订单项级别的折扣拆分（折扣在订单级别整单处理）
- 不改变手动记账的流程（手动记账不受品类影响）

## Decisions

### D1: 品类绑定而非商品绑定

**决策**: 在品类（category）级别绑定账本，而非在商品（product）级别。

**理由**: 用户反馈同一品类的商品总是记账到同一账本，品类级别的绑定减少配置工作量。新商品只需选择品类即可自动继承账本绑定。

**备选方案**:
- 商品级别绑定：每件商品单独绑定，灵活性高但配置量大
- 品类默认 + 商品覆盖：兼顾两者但复杂度高，当前不需要

### D2: 品类作为独立实体

**决策**: 将现有 `product.category`（自由文本）提升为独立实体，新增 `category` 表。

**理由**: 品类需要绑定账本，必须持久化存储。自由文本无法可靠地维护关联关系。

**迁移策略**: `product.category` 字段保留为冗余名称，新增 `category_id` 字段。迁移时不自动关联现有商品，现有商品的 `category_id` 为 `None`，运行时视为"未分类"品类。

### D3: 按比例分摊折扣

**决策**: 折扣/抹零金额按各品类组 subtotal 占总金额的比例分摊，最后一条用补差值。

**理由**: 折扣发生在整单级别（"抹零收1000"），不是逐项谈价。按比例分摊最自然，尾差用补差保证总额精确。

**公式**: `品类分摊折扣 = 折扣总额 × (品类subtotal / 总应收)`，最后一条 = `折扣总额 - 已分摊合计`

### D4: 冲账使用专用 WriteOff 类型

**决策**: 新增 `AccountingType::WriteOff` 枚举变体，冲账金额为负数。

**理由**: 使用专用类型而非反向类型（Income→Expenditure），语义更清晰。金额为负数表示"减少"，统一适用于销售和采购场景。

**冲账记录结构**:
- `accounting_type`: `WriteOff`
- `amount`: 负数（如 `-6.24`）
- `write_off_id`: 指向对应的主记录 id
- `book_id`: 与主记录相同
- `order_id`: 与主记录相同

### D5: 删除订单上的 accounting_record_id

**决策**: 移除 `order.accounting_record_id` 字段，改为通过 `accounting_record.order_id` 反查。

**理由**: 一张订单现在会产生多条记账记录（按品类拆分 + 可能的冲账），单条 FK 不够用。反查方案简洁，且 `accounting_record.order_id` 已存在。

**主记录与冲账区分**:
- 主记录：`write_off_id = None`
- 冲账记录：`write_off_id = Some(主记录id)`

### D6: 订单业务类型枚举设计

**决策**: 新增 `OrderSubType` 枚举，包含 `Wholesale`、`Retail`、`WholesalePurchase`、`PeerTransfer`。

**理由**: 销售和采购共享一个枚举，通过 order_type 约束可选值。创建后不可修改，确保统计数据的一致性。

**默认值规则**:
- Sales + 有客户 → Wholesale
- Sales + 无客户 → Retail
- Purchase → WholesalePurchase

### D7: "未分类"品类自动创建

**决策**: 系统启动时检查是否存在默认品类，不存在则自动创建，名称"未分类"，两个 book_id 都指向 `DEFAULT_BOOK_ID`。

**理由**: 与账本的"未归类账目"模式一致。保证所有商品都有品类归属，避免空值处理。

## Risks / Trade-offs

- **[尾差精度]** 按比例分摊可能出现 Decimal 精度问题 → 使用补差值策略，最后一条用差值而非比例计算，保证总额精确
- **[品类删除]** 删除品类时可能存在关联商品 → 级联策略：将关联商品的 category_id 置为 None（视为未分类），或禁止删除有商品的品类
- **[订单反查性能]** 通过 accounting_record.order_id 反查可能返回多条记录 → 订单结算产生的记录数量有限（品类数 × 2），性能影响可忽略
- **[迁移兼容性]** 删除 order.accounting_record_id 是破坏性变更 → 需要在迁移中处理已结算订单的关联数据
