## Context

当前系统使用 Sea-ORM 的 entity-first 工作流，所有实体定义在 `src-tauri/src/entity/` 目录，数据库初始化时通过 `with_install_entities` 函数同步 schema。系统已存在的 `accounting_record` 和 `accounting_record_seq` 实体提供了 ID 生成的参考实现。记账记录和账本之间需要建立一对多关系，类似附件和记账记录的关系。

## Goals / Non-Goals

**Goals:**
- 提供完整的账本管理功能（CRUD）
- 实现账本删除时关联记录的自动迁移
- 保持与现有记账系统无缝集成
- 确保默认账本的安全性和不可删除性
- 支持分页查询账本列表和账本中的记账记录

**Non-Goals:**
- 账本之间的记录转移功能
- 账本共享或权限管理
- 账本统计数据或报表功能

## Decisions

### 1. 账本 ID 格式选择

**决策**: 采用 `yyyyxxxx` 格式（4位年份+4位流水号）

**理由**:
- 与 `accounting_record` 的 `YYYYMMDDNNNNN` 格式风格一致
- 按年份分组便于查询和管理
- 4位流水号每年可容纳9999个账本，足够个人/小企业使用
- 避免与记账记录 ID 冲突（前者8位，后者13位）

**替代方案**: 全局递增整数
- 问题：无法直观区分年份，不适合账本场景

### 2. 默认账本保护机制

**决策**: 默认账本（id=10000001）设置为系统级保护对象

**理由**:
- 简化删除逻辑，只需检查 id 是否等于 10000001
- 固定 ID 便于跨环境一致性
- 确保未归类记录始终有归属

**替代方案**: 使用 is_system 字段标记
- 问题：增加数据库复杂度，需要额外索引

### 3. 删除迁移的原子性

**决策**: 使用数据库事务确保删除和记录迁移的原子性

**理由**:
- 防止部分失败导致数据不一致
- 遵循现有 `accounting_record_seq` 的实现模式
- Sea-ORM 提供原生事务支持

### 4. 关系定义方式

**决策**: 在 `AccountingRecord` 实体中定义到 `AccountingBook` 的关系

**理由**:
- 查询记账记录时需要关联账本信息
- 便于实现 `find_related` 和 `link` 操作
- 与 Sea-ORM 最佳实践一致

### 5. 分页查询参数设计

**决策**: 使用标准的 page（页码）和 page_size（每页数量）参数进行分页，从 1 开始计数

**理由**:
- 用户友好的分页方式，符合常见习惯
- page 从 1 开始符合用户认知（第一页）
- 简化前端分页组件的实现
- 便于计算总页数（total_pages = ceil(total / page_size)）

**替代方案**: 使用 offset 和 limit 方式
- 问题：对用户不够直观，需要自行计算偏移量

**分页响应结构**:
- `data`: 数据列表
- `total`: 总数量
- `page`: 当前页码
- `page_size`: 每页数量
- `total_pages`: 总页数

## Risks / Trade-offs

### 风险1: 历史记录的 book_id 为 NULL

**缓解措施**: 删除账本时仅迁移 `book_id IS NOT NULL` 的记录，NULL 值记录自动归属未归类账本

### 风险2: 账本名称冲突

**缓解措施**: 当前不强制唯一性，允许同名账本（用户可自行管理）

### 风险3: 批量删除记录的性能问题

**缓解措施**: 删除操作不频繁，使用标准 UPDATE 即可。后续如遇性能问题可考虑批量更新或后台任务

## Migration Plan

1. **新增表结构**: Sea-ORM 会自动创建 `accounting_book` 和 `accounting_book_seq` 表
2. **现有表变更**: `accounting_record` 表自动添加 `book_id` 列
3. **数据迁移**: 现有记录的 `book_id` 默认为 NULL，无需处理
4. **默认账本初始化**: 数据库首次启动时插入默认账本记录
5. **回滚策略**: 若需回滚，需手动删除新增表和列（生产环境需编写迁移脚本）
