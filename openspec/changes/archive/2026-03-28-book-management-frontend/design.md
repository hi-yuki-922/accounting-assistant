## Context

当前项目的账本管理后端接口已经完整实现（创建、查询、更新、删除账本，以及按账本查询记录），包括分页查询和多种筛选条件。但前端缺少对应的管理界面，用户只能通过后端 API 操作账本，无法在应用中直观地查看和管理多个账本。

项目技术栈为 Tauri 2.0（桌面应用）+ React 19 + TypeScript + shadcn/ui。路由使用 @tanstack/react-router，已有侧边栏和底部导航栏布局。

## Goals / Non-Goals

**Goals:**
- 提供直观的账本列表页面，以卡片视图展示所有账本
- 支持创建、编辑、删除账本操作
- 实现账本图标选择功能，提升视觉识别度
- 支持拖拽排序，排序配置持久化存储
- 提供账本详情页面，展示账本内的记账记录
- 支持记录筛选（时间范围、类型、渠道、状态）和分页
- 在导航栏中添加账本入口

**Non-Goals:**
- 账本统计报表（收入/支出汇总、趋势图等）
- 记录的批量操作（批量删除、批量转移）
- 账本导出功能
- 多账本对比分析

## Decisions

### 1. 持久化存储方案

**决策：使用 localStorage 存储账本排序配置**

**理由：**
- 项目已在多个地方使用 localStorage（API Key、模型选择、通知设置），保持一致性
- 排序配置是轻量级数据，localStorage 容量（通常 5-10MB）完全足够
- 无需添加额外依赖（如 @tauri-apps/plugin-store）
- 即使丢失也只是恢复默认排序，不影响数据完整性

**替代方案考虑：**
- **@tauri-apps/plugin-store**: 更可靠的持久化，但需要添加依赖和配置 Rust 端，对于此场景过度设计
- **文件存储**: 完全控制但复杂度高，需要处理权限和错误

### 2. 拖拽排序实现

**决策：使用 dnd-kit (@dnd-kit/core @dnd-kit/sortable @dnd-kit/utilities)**

**理由：**
- 现代化、TypeScript 友好，与 React 19 良好兼容
- 轻量级（~30KB），性能优秀
- 支持触摸操作，在桌面和平板上体验一致
- API 简洁，社区活跃

**替代方案考虑：**
- **react-beautiful-dnd**: 老牌方案，但维护较少且对 React 19 支持有问题
- **react-dnd**: 功能强大但 API 复杂，学习成本高

### 3. 账本记录数量维护策略

**决策：在 AccountingBook 实体中新增 `record_count` 字段，在添加/删除记录时更新**

**理由：**
- 避免每次展示账本列表时都执行 COUNT 查询，提升性能
- 记录数量是高频展示的数据，缓存策略合理
- 在添加/删除记录时同步更新，保证数据一致性

**实现细节：**
- `AccountingService.add_record()`: 创建记录后，更新对应账本的 `record_count += 1`
- `AccountingBookService.delete_book()`: 删除账本时，将所有记录迁移到默认账本，更新默认账本的 `record_count += 迁移数量`

### 4. 默认账本处理

**决策：默认账本（ID: 10000001，"未归类账目"）始终排在列表最后，不能删除**

**理由：**
- 默认账本是系统兜底，保证所有记录都有归属
- 用户体验上，默认账本属于系统行为，不应被用户删除
- 始终排在最后，避免用户误操作或混淆

### 5. 响应式布局方案

**决策：使用 CSS Grid 实现响应式卡片布局**

**理由：**
- Grid 对二维布局控制更灵活，支持指定最小宽度和自动填充
- 实现简单，无需额外依赖
- 适配移动端（1列）、平板（2-3列）、桌面（3-5列）

**实现方式：**
```css
grid-template-columns: repeat(auto-fill, minmax(280px, 1fr));
```

### 6. 图标配置管理

**决策：将图标选项作为独立配置文件，支持后续扩展**

**理由：**
- 集中管理图标选项，便于维护和扩展
- 与业务逻辑解耦，添加新图标只需修改配置文件
- 可未来支持用户自定义图标或图标包

**实现：**
- `src/config/book-icons.ts`: 定义 `BOOK_ICONS` 常量，包含图标 ID、名称、icon 组件名
- `BookIconPicker` 组件渲染配置中的图标选项

### 7. 数据库 Schema 迁移

**决策：使用 Sea-ORM 的 schema sync 功能自动迁移**

**理由：**
- Sea-ORM 自动检测 Entity 变化并同步到数据库
- 无需手动编写 migration SQL，减少出错风险
- 项目已在初始化时使用此机制

## Risks / Trade-offs

### 1. localStorage 数据丢失

**风险**: 用户清除浏览器数据时，账本排序配置会丢失

**缓解措施**:
- 丢失后恢复默认排序（按创建时间），不影响功能
- 在用户设置中增加"重置排序"按钮，让用户主动控制

### 2. record_count 数据不一致

**风险**: 如果直接操作数据库或服务出现异常，`record_count` 可能与实际记录数不一致

**缓解措施**:
- 在创建/删除记录的同一事务中更新 `record_count`（如适用）
- 提供重建 `record_count` 的管理命令（可在未来添加）
- 在账本详情页查询记录时使用实际查询结果，不依赖缓存的 `record_count`

### 3. 拖拽排序性能

**风险**: 账本数量很大时，拖拽排序可能有性能问题

**缓解措施**:
- 个人/小企业使用场景，账本数量通常在几十个以内，不会成为瓶颈
- dnd-kit 使用虚拟滚动优化，性能良好
- 如果未来账本数量超过 100，可考虑分页或搜索优化

### 4. 图标数量控制

**风险**: 图标选项过多会增加选择复杂度

**缓解措施**:
- 初始提供 15-20 个常用图标，覆盖主要使用场景
- 使用分类或搜索功能（如果图标数量超过 30）
- 提供最近使用过的图标快捷选择

## Migration Plan

### 数据库迁移

Sea-ORM 在应用启动时自动检测 Entity 变化并同步 schema：

1. `accounting_book` 表新增 `record_count` 列（i32，默认 0）
2. `accounting_book` 表新增 `icon` 列（varchar，可选）
3. 初始化时扫描现有账本，填充 `record_count`（如有必要）

### 前端部署

前端变更不涉及数据迁移，直接部署新版本即可：
1. 用户首次访问 `/books` 页面时，localStorage 中没有排序配置，使用默认排序
2. 用户拖拽排序后，配置自动保存到 localStorage
3. 回滚到旧版本时，页面不可用，但数据不受影响
