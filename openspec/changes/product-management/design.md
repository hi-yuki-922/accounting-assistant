## Context

当前应用已完成账本管理、记账记录管理和客户管理模块，采用 Entity-First 分层架构（Entity → Service → Command → Frontend）。商品信息管理是订单模块和 Agent 的前置依赖，需要遵循现有架构模式新增商品能力。

现有技术约束：
- Sea-ORM entity-first 模式，无物理外键，通过应用层维护引用完整性
- 实体 ID 采用日期序列模式（`YYYYMMDDNNNNN`），需要配套的 seq 实体
- 服务层接收 `DatabaseConnection`，通过 Tauri `app.manage()` 注入
- 前端使用 `tryCMD<T>` 封装 Tauri invoke，返回 `neverthrow Result`

## Goals / Non-Goals

**Goals:**
- 新增完整的商品信息 CRUD 能力（后端 + 前端）
- 支持商品模糊搜索（覆盖名称、分类、关键词），为 Agent 的自然语言匹配提供基础
- 支持商品分类管理
- 支持参考售价和参考采购价（仅作建议，非固定价格）
- 支持关键词标注，用户可手动维护商品关键词以增强检索能力
- 优化计量单位输入体验，提供常用单位快捷选择
- 完全遵循现有分层架构和编码规范

**Non-Goals:**
- 不实现库存管理（Phase 2）
- 不实现客户定价表（Phase 2）
- 不实现商品图片/附件管理
- 不实现价格历史追踪
- 不涉及订单或 Agent 功能（Phase 1B/1C）

## Decisions

### 1. 商品实体字段设计

商品价格采用双参考价设计（`default_sell_price` + `default_purchase_price`），均为可空字段。理由：
- 支持价格波动频繁的商品场景（如海鲜），参考价可随时更新
- 允许部分商品不设参考价（新品或价格不固定的商品）
- 真实成交价存储在订单明细中，商品价格仅作建议

**替代方案**：维护商品价格历史表 → 过度设计，当前阶段不需要价格趋势分析。

### 2. 商品分类采用简单字符串字段

使用 `category: Option<String>` 而非独立的分类实体或枚举。理由：
- 小型商户的商品分类简单且可能频繁调整
- 避免过度设计，后续如需结构化分类可迁移
- 搜索时通过 LIKE 模糊匹配即可

**替代方案**：使用 `ProductCategory` 枚举 → 不够灵活，新增分类需要改代码。使用独立分类表 → 当前阶段过重。

### 3. 单位采用字符串字段 + Chips 快捷选择

使用 `unit: String` 存储计量单位，前端提供常用单位 Chips 快捷选择。理由：
- 商户使用的计量单位多样且可能自定义（斤、个、箱、盒、桶、包...）
- 不需要枚举约束，保持灵活性
- Chips 快捷选择覆盖常见单位（斤、公斤、个、件、箱、盒、袋、瓶、包），降低输入成本
- 用户仍可自由输入不在预设列表中的单位

**预设常用单位列表**: 斤、公斤、个、件、箱、盒、袋、瓶、包（硬编码常量，后续可按需扩展）

### 4. 商品名称唯一性

商品名称不做数据库层唯一约束，由应用层在创建时检查。理由：
- 同名商品可能存在不同规格（如"苹果"可能是不同品种）
- 保持与现有实体一致的模式（现有实体均无唯一约束）
- Agent 可通过名称 + 分类 + 单位组合来识别商品

### 5. 删除策略

商品删除采用软删除判断：如果商品被订单明细引用则禁止删除。当前阶段只实现硬删除 + 引用检查（订单模块实现后生效）。暂时允许直接删除，因为订单模块尚未实现。

### 6. 关键词字段设计

使用 `keywords: Option<String>` 存储关键词，多个关键词以分号（`;`）分隔。理由：
- 纯文本存储简单可靠，SQLite LIKE 直接支持模糊搜索
- 分号分隔符直观，不依赖 JSON 解析
- 关键词由用户手动维护，不涉及 AI 自动生成（当前阶段）
- 前端使用 Tag 输入模式，提交时拼接为分号分隔字符串，编辑时解析回 Tag 列表

**示例**: 商品"六头鲍鱼"，SKU "35#"，关键词存储为 `"六头;六粒;35#"`

**替代方案**: JSON 数组存储（`["六头","六粒","35#"]`）→ 增加序列化/反序列化复杂度，LIKE 搜索同样可行，但收益不明显。

### 7. 前端路由与页面结构

新增 `/products` 路由，采用与 `/customers` 一致的页面结构：
- `routes/products.tsx` — 布局（AppLayout + Outlet）
- `routes/products.index.tsx` — 商品列表页
- 页面组件：商品卡片、创建/编辑对话框、删除确认对话框

## Risks / Trade-offs

- **[无物理外键]** → 由应用层保证引用完整性。删除商品前需检查是否被引用，当前阶段订单模块未实现，暂时无引用检查。
- **[分类为自由文本]** → 可能出现"水果"和"水果类"等不一致分类。后续可通过下拉建议（基于已有分类值）缓解。
- **[参考价可空]** → Agent 在建议价格时需要处理无参考价的场景，应提示用户输入。
- **[关键词为手动维护]** → 关键词质量和覆盖度依赖用户主观判断，可能出现遗漏。后续可考虑 AI 辅助建议。
